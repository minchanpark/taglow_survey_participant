# Taglow Survey Participant PRD v8

# 1. 문서 개요

## 1.1 문서 목적

본 문서는 Taglow Survey의 **참여자 View** 요구사항을 정의한다.

Taglow Survey는 자치회·학생기구가 관리자 페이지에서 섹션별 설문을 만들고, 설문을 공개하면 URL/QR이 생성되며, 참여자가 해당 URL로 접속해 로그인 후 설문에 응답하는 서비스이다.

참여자 View의 목표는 다음과 같다.

1. 참여자가 URL/QR만으로 쉽게 설문에 진입할 수 있게 한다.
2. 참여자가 Google 소셜 로그인으로 본인 인증을 완료하게 한다.
3. `@handong.ac.kr` 계정만 설문에 참여할 수 있게 한다.
4. 섹션별로 묶인 질문을 모바일에서 부담 없이 응답하게 한다.
5. 만족도, 중요도, 주관식, 복수응답, 순위형, 이미지 태깅 등 다양한 질문 유형을 쉽게 입력하게 한다.
6. 응답 도중 이탈해도 프론트엔드 캐시 기반으로 임시 저장/복구가 가능하게 한다.
7. 한국어/영어 문구를 참여자 언어 설정에 맞춰 표시한다.
8. 관리자 페이지에서 기본 정보 필터링, 섹션별 분석, 이미지 태깅 히트맵, 주관식 분석을 할 수 있도록 구조화된 데이터를 수집한다.

## 1.2 전체 운영 맥락

```text
관리자 로그인
↓
관리자 설문 생성
↓
섹션 생성
↓
섹션 안에 질문 생성
↓
다국어 문구 입력
↓
이미지 태깅용 공간 이미지 업로드
↓
설문 공개
↓
공개 URL/QR 생성
↓
참여자가 URL/QR 접속
↓
Google 로그인
↓
@handong.ac.kr 검증
↓
섹션별 설문 응답
↓
프론트엔드 캐시에 임시 저장
↓
최종 제출
↓
responses + answers 저장
↓
관리자 페이지에서 분석
```

# 2. 제품 목표

## 2.1 참여자 View 핵심 목표

참여자 페이지는 “학생이 쉽게 좋은 데이터를 남기는 화면”이어야 한다.

좋은 데이터란 다음을 의미한다.

- 기본 정보가 누락되지 않는다.
- 경험한 항목에 대해서만 만족도/중요도를 응답한다.
- 불만족 이유가 구조화되어 있다.
- 공간 문제는 위치 정보와 함께 남는다.
- 주관식 원문은 분석 가능한 카테고리와 연결된다.
- 복수응답/순위형 응답은 관리자 분석에 바로 쓰일 수 있는 값으로 저장된다.

## 2.2 참여자 UX 원칙

1. 첫 화면에서 설문 목적과 예상 부담을 줄인다.
2. 모든 질문을 한 번에 보여주지 않고 섹션 단위로 나눈다.
3. 한 화면에서 하나의 명확한 응답 작업만 수행하게 한다.
4. 가능한 모든 입력은 선택형으로 제공한다.
5. 자유 입력은 마지막 보완 수단으로 사용한다.
6. 경험하지 않은 항목은 빠르게 건너뛰게 한다.
7. 낮은 만족도일 때만 후속 이유를 묻는다.
8. 이미지 태깅은 쉬운 문구와 명확한 UI로 제공한다.
9. 진행률과 남은 작업량을 계속 보여준다.
10. 제출 완료 후 “내 의견이 반영된다”는 피드백을 제공한다.

# 3. 범위 정의

## 3.1 MVP 포함 범위

### 접근/인증

- 공개 URL 접속
- QR 접속
- 설문 상태 확인
- Google 소셜 로그인
- `@handong.ac.kr` 도메인 검증
- 비허용 도메인 접근 차단
- 중복 제출 제한

### 설문 응답

- 설문 안내 화면
- 언어 선택 또는 기본 언어 렌더링
- 섹션 목록 및 진행률
- 기본 정보 입력
- 경험 여부 응답
- 만족도/중요도 5점 척도
- 낮은 만족도 후속 질문
- 단일 선택
- 복수 선택
- 순위형 선택
- 주관식 입력
- 이미지 태깅
- 주의력 체크 문항
- 제출 전 검토
- 제출 완료 화면

### 데이터 저장

- 최종 제출 시 `responses` 생성
- 기본 정보 컬럼 저장
- 모든 질문 응답을 `answers`에 저장
- 이미지 태깅 좌표를 `x_ratio`, `y_ratio`로 저장
- 원본 제출 payload 저장
- 참여자 선택 언어 저장

### 임시 저장

- 서버 저장이 아닌 프론트엔드 캐시 기반 draft 저장
- 같은 설문 + 같은 사용자 기준 draft 분리
- 제출 성공 시 draft 삭제
- 브라우저/기기 변경 시 draft 복구는 지원하지 않음

## 3.2 MVP 제외 범위

- 학교 포털 SSO
- 서버 기반 실시간 draft 저장
- 참여자 간 응답 공유
- 댓글/대댓글
- 동영상 태깅
- 다중 기기 간 draft 동기화
- 장기 사용자 프로필 관리
- 참여자별 응답 이력 페이지

## 3.3 후순위 고려 기능

- 서버 기반 임시 저장
- 참여자 후속 인터뷰 신청
- 응답 완료 후 리워드 코드 발급
- 음성 응답 입력
- 사진 첨부형 응답
- 참여자 언어 자동 감지 고도화

# 5. 로그인 및 접근 제어

## 5.1 요구사항

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| P-AUTH-01 | Google 로그인 | 참여자는 Google 계정으로 로그인한다. | Must |
| P-AUTH-02 | 도메인 검증 | `@handong.ac.kr` 이메일만 허용한다. | Must |
| P-AUTH-03 | 서버 검증 | 프론트엔드 표시뿐 아니라 API에서 최종 검증한다. | Must |
| P-AUTH-04 | 비허용 안내 | 허용되지 않은 계정은 접근 불가 사유를 안내한다. | Must |
| P-AUTH-05 | 중복 제출 제한 | 동일 사용자 기준 중복 제출을 제한한다. | Must |
| P-AUTH-06 | 로그인 후 원래 URL 복귀 | 로그인 후 접속하던 설문으로 돌아온다. | Must |

## 5.2 접근 불가 문구 예시

```text
이 설문은 한동대학교 계정(@handong.ac.kr)으로만 참여할 수 있습니다.
학교 Google 계정으로 다시 로그인해주세요.
```

## 5.3 중복 제출 문구 예시

```text
이미 제출한 설문입니다.
제출 내용 수정이 필요한 경우 담당자에게 문의해주세요.
```

# 6. 다국어 렌더링

## 6.1 목적

관리자가 한국어/영어 문구를 입력하면 참여자 페이지는 선택 언어에 맞는 문구를 보여준다.

## 6.2 요구사항

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| P-LANG-01 | 언어 선택 | 설문 시작 전 한국어/영어를 선택할 수 있다. | Should |
| P-LANG-02 | 기본 언어 적용 | 언어 선택이 없으면 설문 기본 언어를 적용한다. | Must |
| P-LANG-03 | 섹션/질문 번역 렌더링 | title_ko/title_en 또는 translations 값을 사용한다. | Must |
| P-LANG-04 | 선택지 번역 렌더링 | 선택지도 언어별 label을 표시한다. | Must |
| P-LANG-05 | fallback | 번역이 없으면 기본 언어 문구를 표시한다. | Must |
| P-LANG-06 | 응답 저장 언어 | 참여자가 사용한 language를 responses에 저장한다. | Should |

## 6.3 저장 원칙

```text
문구는 언어별로 표시한다.
응답은 언어별로 따로 저장하지 않는다.
```

예시:

```text
사용자 화면: “How satisfied are you with the laundry room?”
저장 데이터: question_id, score_value, metric_type, topic_key
```

# 7. 설문 시작 화면

## 7.1 목적

첫 화면은 참여자의 부담을 줄이고 설문의 목적을 설명해야 한다.

## 7.2 표시 요소

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| P-START-01 | 설문 제목 | 관리자 입력 제목을 표시한다. | Must |
| P-START-02 | 설문 목적 | 응답이 어떻게 활용되는지 설명한다. | Must |
| P-START-03 | 예상 소요 시간 | 예상 시간을 표시한다. | Must |
| P-START-04 | 로그인 안내 | Google 로그인과 도메인 제한을 안내한다. | Must |
| P-START-05 | 응답 방식 안내 | 선택, 짧은 의견, 사진 위치 표시 등 방식을 안내한다. | Should |
| P-START-06 | 개인정보/익명 안내 | 수집 정보와 활용 범위를 안내한다. | Must |
| P-START-07 | 시작하기 버튼 | 로그인 또는 기본 정보 입력으로 이동한다. | Must |

## 7.3 예시 문구

```text
생활관 정기 설문조사

약 7~10분 정도 소요됩니다.
응답은 생활관 시설 개선과 자치회 사업 기획에 활용됩니다.
내가 경험한 항목만 답하면 됩니다.
시설 관련 의견은 사진 위에 직접 표시할 수 있습니다.
```

# 8. 섹션 기반 응답 구조

## 8.1 목적

질문이 많아도 참여자가 길고 복잡하게 느끼지 않도록 섹션별로 나누어 응답하게 한다.

## 8.2 섹션 화면 요구사항

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| P-SEC-01 | 섹션 제목 표시 | 현재 섹션 제목을 표시한다. | Must |
| P-SEC-02 | 섹션 설명 표시 | 관리자가 입력한 설명을 표시한다. | Should |
| P-SEC-03 | 섹션 진행률 | 현재 섹션 내 완료 질문 수를 표시한다. | Must |
| P-SEC-04 | 전체 진행률 | 전체 설문에서 현재 위치를 표시한다. | Must |
| P-SEC-05 | 이전/다음 이동 | 섹션 간 이동을 지원한다. | Must |
| P-SEC-06 | 필수 응답 검증 | 필수 문항 미응답 시 다음 단계로 넘어갈 수 없다. | Must |
| P-SEC-07 | 임시 저장 상태 표시 | 자동 저장됨/저장 중/복구 가능 상태를 표시한다. | Should |

```text
기본 정보

세탁기 및 건조기
기타 생활
글로벌 라운지
제출자 정보
```

# 9. 기본 정보 입력

## 9.1 목적

관리자 분석의 Global Filter Bar에 필요한 기본 정보를 수집한다.

## 9.2 수집 항목

| 항목 | 저장 위치 | 응답 방식 | 필수 여부 |
| --- | --- | --- | --- |
| 성별 | responses.gender | 칩 선택 | Must |
| 학기 | responses.semester_group | 칩/드롭다운 | Must |
| 학부 | responses.department | 검색형 드롭다운 | Must |
| 소속 RC | responses.rc | 칩 선택 | Must |
| 거주 생활관 | responses.dormitory | 칩 선택 | Must |
| 인실 | responses.room_type | 칩 선택 | Must |
| 생활관 거주 경험 | responses.dorm_experience | 칩 선택 | Should |

## 9.3 UX 요구사항

- 가능한 한 한 화면 또는 두 화면 안에 입력한다.
- 모든 값은 선택형으로 제공한다.
- 학부처럼 선택지가 많은 경우 검색형 드롭다운을 제공한다.
- 제출 전 수정이 가능해야 한다.
- 기본 정보는 개인 식별이 아니라 분석용 분류 정보임을 안내한다.

# 10. 경험 여부 응답

## 10.1 목적

경험하지 않은 항목에 만족도를 묻지 않음으로써 피로도와 부정확한 응답을 줄인다.

## 10.2 요구사항

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| P-EXP-01 | 경험 여부 질문 | 섹션 또는 항목별 경험 여부를 묻는다. | Must |
| P-EXP-02 | 경험자만 후속 질문 | 경험 있음인 경우에만 만족도/중요도 질문을 보여준다. | Must |
| P-EXP-03 | 미경험 이유 | 미이용 이유를 선택형으로 받을 수 있다. | Should |
| P-EXP-04 | 처음 들음 저장 | 인지도 부족 분석용으로 저장한다. | Should |
| P-EXP-05 | 해당 없음 제외 | 분석 대상에서 제외한다. | Must |

## 10.3 화면 예시

```text
이번 학기 오피스 아워를 이용해본 적이 있나요?

- 이용해봤다
- 들어봤지만 이용하지 않았다
- 처음 들어봤다
- 해당 없음
```

## 10.4 저장 예시

```json
{
  "experience_status": "heard_not_used",
  "no_experience_reason": "필요하지 않았음"
}
```

# 11. 만족도/중요도 응답

## 11.1 목적

관리자가 평균 점수, Gap, Borich, Locus 분석을 할 수 있도록 점수 데이터를 수집한다.

## 11.2 요구사항

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| P-SCALE-01 | 5점 척도 | 만족도/중요도를 1~5점으로 입력한다. | Must |
| P-SCALE-02 | 모바일 친화 UI | 버튼/별점/칩 형태로 빠르게 선택한다. | Must |

# 12. 낮은 만족도 후속 질문

## 12.1 목적

만족도가 낮을 때만 이유를 묻는다. 이렇게 해야 참여자 피로도를 줄이고 불편 이유를 구조화할 수 있다.

## 12.2 조건부 노출

기본 기준은 다음과 같다.

```text
만족도 1~2점: 후속 질문 필수
만족도 3점: 후속 질문 선택
만족도 4~5점: 후속 질문 생략
```

관리자 설정에 따라 기준은 달라질 수 있다.

## 12.3 요구사항

| ID | 기능 | 설명 | 우선순위 |
| --- | --- | --- | --- |
| P-LOW-01 | 낮은 만족도 이유 선택 | 불편 이유를 선택형으로 받는다. | Must |
| P-LOW-02 | 복수 선택 | 여러 이유를 고를 수 있다. | Should |
| P-LOW-03 | 한 줄 설명 | 필요한 경우 짧은 설명을 입력한다. | Should |
| P-LOW-04 | 공간 연결 | 공간 문제라면 이미지 태깅으로 이어질 수 있다. | Should |
| P-LOW-05 | 심각도 | 문제의 심각도를 간단히 선택한다. | Should |

## 12.4 예시

```text
낮은 점수를 준 이유를 선택해주세요.

- 기기 수가 부족함

- 청소 상태가 좋지 않음
```

# 29. 결론

Participant PRD v8의 핵심은 다음이다.

> 참여자는 URL/QR로 들어와 Google 로그인 후 섹션별로 설문에 참여하고, 본인이 경험한 항목만 빠르게 응답하며, 낮은 만족도 이유와 공간 위치 의견을 쉽게 남길 수 있어야 한다.

참여자 화면의 목표는 단순히 많은 질문을 보여주는 것이 아니다.

```text
쉽게 들어오게 한다.
빠르게 이해하게 한다.
해당되는 질문만 답하게 한다.
좋은 데이터를 남기게 한다.
중간 이탈을 줄인다.
제출 후 의견이 반영된다는 느낌을 준다.
```

DB는 단순 구조를 유지한다.

```text
surveys
survey_sections
questions
survey_assets
responses
answers
```

참여자 페이지는 이 구조를 기반으로 섹션과 질문을 렌더링하고, 최종 제출 시 `responses`와 `answers`에 분석 가능한 데이터를 저장한다.
